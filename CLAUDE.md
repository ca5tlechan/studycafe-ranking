## 0. 이 개정본에서 바뀐 것 (2차 리뷰 반영)

초안 대비 아래 항목이 확정/수정되었다. 상세는 각 섹션 참조.

1. **하루 마감 방식 = 04:00 일일 자동 마감 + 사전 알림 + 경고/페널티** (§3.6 전면 개정). 초안의 "단순 최대 세션 캡"을 대체.
2. **최소 세션(10분) 필터는 랭킹 쿼리(HAVING)가 아니라 집계 단계에서 적용** — 초안 §6 SQL의 의미 오류 수정 (§3.6d, §6).
3. **집계 멱등성 확보**: `daily_study_records`는 `+=`가 아니라 **영향받은 스터디 날짜를 원본 세션에서 재계산해 SET** (§3.1, §4).
4. **세션이 닫히는 모든 경우(수동 체크아웃 + 04:00 자동 마감 + 강제 종료)에 분할·집계를 태운다** (§3.1, §6).
5. **이름 마스킹 함수의 1글자 이하 방어** 추가 (§3.3).
6. **랭킹 시간 캡 = 하루 16h + 주 84h** — 랭킹에만 적용, 마이페이지는 실제 시간 그대로, 초과분만 제외(=A안) (§3.6e).
7. **학교 랭킹 분모 = 선택 기간의 활동 인원**, **최소 인원 5명**, **동점 = 활동 인원 많은 학교 상위** (§3.4, §3.5).
8. **랭킹 이름 표기 = `김O현(ㅁㅁ중)`** 형식, 무소속은 `(무소속)`, 같은 학교 동명이인은 접미 숫자 (§3.3, §5.2).
9. **"우리 학교 페이지" 신설** — 같은 학교 내 개인 공부시간 랭킹 (§5.3).
10. **기간 내 기록이 없으면 "순위 없음/기록 없음"** 표기 (§5.2).
11. **동시 활성 세션 원천 차단**(부분 유니크 인덱스) 등 안정성 개선 (§4, §6).
12. 프론트 = **React + Vite 확정**. 학교 소속 검증은 하지 않음(큐레이션 목록으로 대응).

## 1. 프로젝트 개요

스터디카페 이용 시간을 기록하고, **월/주/기간 단위로 학생 개인 및 학교별 공부 시간 랭킹**을 보여주는 앱이다. 특정 챌린지 스터디카페 한 곳(**24시간 운영**)을 대상으로 파일럿을 진행한다.

- **[확정] 플랫폼: PWA** (iOS Safari / Android Chrome 공통, 앱스토어 미경유, 웹 URL 배포)
- 이 선택에 따른 **기술적 귀결**(중요):
  - **체크인/체크아웃은 QR 촬영 방식만 사용한다.** iOS Safari는 Web NFC를 지원하지 않으므로 NFC는 배제한다.
    - **[예외 · 확정] 개발 빌드 한정 수동 토큰 입력**: 카메라는 보안 컨텍스트(HTTPS/localhost)에서만 열리고 개발용 브라우저·데스크톱에는 카메라가 없어, 이 예외가 없으면 체크인 화면을 실제로 눌러 검증할 수단이 사라진다. 그래서 QR 토큰을 직접 입력해 토글하는 UI를 체크인 화면에 둔다.
      - `import.meta.env.DEV` 로 게이팅한다. **프로덕션 빌드에는 포함되지 않는다** — 번들에서 제거됨을 확인했다(§7 Phase 7).
      - **프로덕션에서 체크인/체크아웃을 발생시키는 경로는 카메라 QR 스캔뿐이다.** 이 불변식이 깨지면(예: 개발 모드로 배포) 원격 체크인이 가능해지므로, 배포 파이프라인은 프로덕션 모드 빌드를 사용해야 한다 (§7 Phase 12).
  - **백그라운드 자동 체크아웃(지오펜싱)은 구현 불가**하다. 웹 표준에 지속적 백그라운드 위치 감지 API가 없다.
  - 체크아웃 누락 대응은 **04:00 일일 자동 마감 + 경고/페널티**로 처리한다 (§3.6).
  - **푸시 알림(Web Push)은 전달이 보장되지 않는다.** iOS는 홈 화면에 설치한 PWA + iOS 16.4+에서만 동작 (§8). 03:30 사전 알림은 "베스트 에포트"이며, 페널티 로직은 알림 수신 여부와 무관하게 세션 상태만으로 판정한다.
- 근본적 한계(정직하게 인지): "그 자리에 물리적으로 있었다"는 것과 "공부했다"는 것은 다르며, 어떤 방식으로도 완전 검증은 불가능하다. 랭킹은 태생적으로 어뷰징 유인이 있으므로 "완벽 차단"이 아니라 "왜곡을 줄이는 장치"를 목표로 한다.

## 2. 기술 스택

### 백엔드
- **[확정] Java + Spring Boot**, REST API
- **[확정] 인증: JWT** (회원가입/로그인 시 토큰 발급, 보호된 엔드포인트는 JWT 필요)
- 의존성(초기): Spring Web, Spring Data JPA, Spring Security, Validation, PostgreSQL Driver
- 스케줄러: Spring `@Scheduled` (04:00 자동 마감, 03:30 사전 알림 배치)
- 에디터: **IntelliJ** (Claude Code JetBrains 플러그인 사용)

### 프론트엔드
- **[확정] React + Vite** (SPA + PWA). 별도 Spring API를 붙이는 클라이언트 SPA 구조라 Vite 빌드가 최적이며, html5-qrcode/차트/vite-plugin-pwa 생태계가 React에 1급으로 붙는다. (Next.js는 SSR 기능이 남아돌아 과함.)
- PWA 설정: `vite-plugin-pwa` (manifest, service worker). Web Push 구현 시 service worker에서 `push` 이벤트 처리.
- 차트: Recharts 또는 Chart.js (택1)
- QR 스캔: `html5-qrcode` 등 카메라 기반 라이브러리
- 에디터: **VS Code** (Claude Code VS Code 확장 사용)
- 실제 UI 구현 단계에서는 별도 디자인 가이드(frontend-design skill 등)를 참조해 완성도를 높인다.

### 데이터베이스
- **[설계] PostgreSQL** (JPA 연동, 랭킹 집계 SQL이 핵심)

### 인프라 / 배포
- **[설계] 개발 단계는 100% 로컬**: Spring Boot는 localhost, DB는 로컬 Postgres(또는 Docker), 프론트는 Vite dev server.
- 실제 배포처(호스팅/도메인)는 **개발 종료 시점에 최신 조건을 재확인해 결정**한다. 지금은 "Spring Boot + 표준 Postgres" 구조만 유지하면 되고, 코드 변경 없이 커넥션 정보만 바꿔 이전 가능하도록 만든다.
- 무료/소액 지향. 무료 티어는 "유휴 시 슬립 → 첫 요청 지연" 트레이드오프가 공통으로 따라온다는 점을 인지한다. (04:00 자동 마감 같은 스케줄 배치가 슬립 인스턴스에서 정확히 도는지 배포 단계에서 반드시 확인.)
- **HTTPS 필수**: 카메라(QR), PWA 설치, Web Push가 보안 컨텍스트에서만 동작. 로컬은 `localhost` 예외 허용.

### 형상관리
- **[확정] GitHub 사용.** **public(오픈소스) 레포**로 운영한다. (초기엔 private 예정이었으나, CodeRabbit AI 코드리뷰를 무료로 쓰기 위해 public으로 전환. 시크릿을 절대 커밋하지 않으므로 공개해도 안전.)
- `.gitignore`에 `.env`, `application-local.yml`, `application-secret*.yml` 등 비밀정보 파일을 반드시 포함. DB 접속정보·JWT 시크릿·(Web Push 사용 시)VAPID 키는 환경변수/`.env`로만 주입하고 커밋하지 않는다. (커밋되는 `application.yml`은 `${...}` 플레이스홀더만.)
- **Phase마다 feature 브랜치 → PR → 머지** 흐름으로 간다(예: `feat/phase-1-auth`). 각 PR은 **CodeRabbit**(`.coderabbit.yaml`)이 보안/구조 중심으로 자동 리뷰한다. **커밋 단위를 작게, 메시지를 명확하게.**

## 3. 핵심 도메인 규칙 (⚠️ 구현 정확도의 핵심)

이 섹션의 규칙들이 이 앱의 난이도와 정확성을 좌우한다. 특히 §3.1은 반드시 단위 테스트로 검증한다.

> **시간대 처리 원칙**: 모든 시각은 `timestamptz`(UTC 저장). 스터디 날짜/주/시간 경계 계산은 전부 **KST(고정 +9, 한국은 DST 없음)** 기준으로 한다. 이 변환은 유틸 하나(`StudyClock` 등)로 캡슐화하고 테스트에 시간대 케이스를 포함한다. §3.1의 "다음 04:00 정각"도 KST 기준이다.

### 3.1 [확정] 하루의 기준은 새벽 4시 — 세션 분할

스터디카페의 하루는 **04:00(KST)에 시작해 다음 날 04:00에 끝난다.**

- 어떤 시각 `T`의 **스터디 날짜(study date)** = `(T - 4시간)`의 날짜.
  - 예: `9일 02:00` → `8일 22:00` → 스터디 날짜 = **8일**
  - 예: `9일 04:00` (정각) → `9일 00:00` → 스터디 날짜 = **9일** (04:00은 새 날의 시작)
  - 예: `9일 05:00` → `9일 01:00` → 스터디 날짜 = **9일**
- 체크인~체크아웃 구간이 04:00을 걸치면 **04:00 지점에서 쪼개서** 각 스터디 날짜에 나눠 귀속한다.

**분할 로직의 위상(중요):**
정상 운영에서는 **04:00 일일 자동 마감(§3.6a)** 때문에 세션이 04:00을 넘지 않는다(자동 마감이 `check_out_at`을 그날 04:00 정각으로 세팅). 따라서 분할은 대부분 no-op이다. 그러나 **배치 지연/미실행, 예외적 지연 종료** 등으로 세션이 04:00을 걸칠 수 있으므로 **분할 로직은 정확성 안전망으로 유지**한다. → **세션이 닫히는 모든 경우(수동 체크아웃 / 04:00 자동 마감 / 강제 종료)에 분할→집계를 태운다.**

**검증용 예시 (반드시 이 케이스로 단위 테스트 — 순수 함수로 테스트):**
입실 `8일 18:00` → 퇴실 `9일 06:00` (총 12시간)
- `8일 18:00 ~ 9일 04:00` = **10시간 → 8일**
- `9일 04:00 ~ 9일 06:00` = **2시간 → 9일**

**분할 알고리즘 (의사코드):**
```
function splitByStudyDate(checkInAt, checkOutAt):   // 모든 계산 KST
    segments = []
    cursor = checkInAt
    while cursor < checkOutAt:
        nextBoundary = cursor 이후 첫 04:00 정각(KST)
        segmentEnd  = min(nextBoundary, checkOutAt)
        studyDate   = (cursor - 4h).toDate()
        segments.add({ studyDate, seconds: segmentEnd - cursor })
        cursor = segmentEnd
    return segments
```

**적용 대상**: 캘린더, 요일별 통계, 주간·월간 합계, 개인·학교 랭킹 (= 시간 "합계"를 다루는 모든 곳).
**미적용 대상**: 24시간 시간대별 히스토그램 (§3.2).

**집계 반영 방식 (멱등성 필수)**: 세션이 닫힐 때 그 세션이 영향을 준 스터디 날짜(들)에 대해 **원본 세션(`check_in_sessions`)에서 해당 유저·해당 날짜의 유효 세션을 전량 재계산해 `daily_study_records.total_seconds`를 SET(덮어쓰기)** 한다. `+=`(증가) 방식은 재처리·재시도 시 **이중 집계**로 시간이 뻥튀기되므로 금지한다. (파일럿 규모에선 재계산 비용 무시 가능.) "유효 세션"의 정의는 §3.6d(최소 10분) 필터를 통과한 세션이다.

### 3.2 [확정] 주(week) 경계도 04:00 기준

- 한 주 = **월요일 04:00 ~ 다음 월요일 04:00**. 스터디 날짜 기준으로 일관되게 처리한다.
- 월(month)/년(year)도 스터디 날짜 기준으로 판단한다.
- 실무적으로 `daily_study_records.study_date`(이미 04:00 보정된 date)에 대한 단순 날짜 범위 필터로 구현된다. (예: 이번 주 = 이번 스터디주의 월~오늘.)

### 3.3 [확정] 이름 마스킹

`display_name`은 실명이며, **랭킹 표시 시 마스킹**한다.

```
function maskName(name):
    n = name.length
    if n <= 1: return name          // 방어: 0~1글자는 그대로 (드문 예외)
    if n == 2: return name[0] + "O"            // 김민 → 김O
    else:      return name[0] + "O"*(n-2) + name[n-1]
                                                // 김O현, 김OO현, 김OOO현
```
- 규칙: **첫 글자와 마지막 글자는 노출, 그 사이는 전부 `O`.** 단 2글자는 예외적으로 "첫 글자 + O", 1글자 이하는 원문 그대로.
- **랭킹 표기 형식** = `마스킹이름(학교축약명)`. 예: `김O현(ㅁㅁ중)`. 무소속은 `김O현(무소속)`.
  - 학교 축약명은 `schools.short_name`을 사용(없으면 `name`).
  - **같은 학교 동명이인**(실명 완전 동일 + 같은 `school_id`)은 나중에 가입한 사용자에게 접미 숫자를 붙인다. 예: `김O현(ㅁㅁ중)2`. (`users.name_seq`로 관리, 첫 사용자=1은 미표기.)
  - 참고: 실명이 달라도 마스킹이 같아질 수 있다(예: 김민현·김서현 → 둘 다 `김O현`). 이 경우는 접미 숫자를 붙이지 않는다(프라이버시상 의도된 동작).
- 마스킹 케이스(2/3/4글자 + 1글자 이하 방어 + 동명이인 접미)를 단위 테스트에 포함한다.

### 3.4 [확정] 학교 랭킹은 "평균"

- 학교 순위는 소속 학생들의 공부 시간 **합계가 아니라 평균**으로 매긴다 (재학생 수 많은 학교의 구조적 이점 제거).
- **[확정] 평균의 분모 = 선택한 기간의 활동 인원** = 그 기간에 (최소 세션 필터 통과) 공부 기록이 **1건 이상 있는 학생 수**. 가입만 하고 안 온 휴면 계정으로 인한 왜곡을 방지한다.
- **[확정] 최소 인원 기준 = 5명.** 위 "활동 인원"이 5명 미만인 학교는 해당 기간 학교 랭킹에서 **제외**한다. (소수 인원의 극단적 왜곡 방지. 값은 §12 조정 가능.)
- **[확정] 랭킹에 인원수도 함께 표시** (예: `서울대학교 · 24명 · 평균 18시간 30분`). 여기서 24명 = 그 기간 활동 인원.

### 3.5 [확정] 동점 처리

- **개인 랭킹**: 공부 시간이 같으면 **해당 기간 내 마지막 체크아웃 시각이 더 이른 쪽이 상위**. (목표 시간을 먼저 채우고 더 늘리지 않은 사람이 위.) `last_checkout_in_period` = 기간 내 유저별 `MAX(check_out_at)`.
- **학교 랭킹**: 평균이 같으면 **활동 인원이 더 많은 학교가 상위**(같은 평균을 더 많은 인원으로 유지 = 더 견고). 그래도 같으면 **학교명 오름차순**으로 최종 확정(결정성 보장).
  - 개인의 "마지막 체크아웃" 규칙을 학교에 그대로 옮기지 않는 이유: 학교 대표값이 밤샘한 한 명 때문에 정해져 의미가 왜곡되기 때문.
- 학교 평균은 초 단위 소수라 완전 동점이 사실상 발생하지 않는다 → 학교 tie-break는 결정성 보장용 안전장치 수준이다.

### 3.6 [확정] 세션 마감 · 어뷰징 방지

체크아웃 누락/조작으로 시간이 부풀지 않게 서버에서 방어한다. **핵심은 04:00 일일 자동 마감이며, 이것이 사실상 "세션 상한" 역할을 겸한다.**

**(a) 04:00 일일 자동 마감 [확정]**
- 매일 **04:00(KST)** 배치가 그 시점 `ACTIVE` 세션을 전부 자동 종료한다. `status = AUTO_CLOSED`, **`check_out_at = 그날 04:00 정각**(배치 실제 실행 시각과 무관하게 04:00으로 세팅). 종료 시 §3.1 분할→집계를 태운다.
- 이로써 정상 운영에서 **어떤 세션도 04:00을 넘지 않고**, 세션은 자연히 하루(<24h) 안에 묶인다. → 계산 단순화 + 미체크아웃 세션의 상한 역할.
- **계속 공부하려면 04:00 이후 다시 체크인**해야 한다(새 스터디 날짜의 새 세션). 이 안내는 앱 체크인 화면에 상시 노출한다.

**(b) 03:30 사전 알림 [확정]**
- 매일 **03:30(값 튜닝 가능, 03:00~03:30)** 배치가 그 시점 `ACTIVE` 유저에게 공지성 알림을 보낸다: "곧 04:00에 하루가 마감됩니다. 계속 공부하시려면 **04:00 이후 다시 체크인**하세요. 체크아웃/재체크인 없이 04:00을 넘기면 자동 종료되고 **경고가 누적**됩니다."
- 전송은 Web Push(설치 PWA 한정, §8). 미설치/구형 iOS에는 도달하지 않을 수 있으므로 **베스트 에포트**로 취급.

**(c) 경고 / 페널티 [확정]**
- 04:00에 **자동 마감된**(= 사용자가 스스로 체크아웃/재체크인 관리를 하지 않은) 세션의 소유자에게 **경고 1회 누적**.
- 누적 **N회(기본 3)** 도달 시 **페널티**. 페널티 기본 동작 = **해당 주(또는 쿨다운 기간) 랭킹에서 제외 + 프로필에 경고 표시**. (구체 강도는 §12에서 조정.)
- 경고는 **매월 1일(스터디 기준) 리셋**한다(영구 누적 금지).
- **공정성 주의(중요)**: 03:30 알림 도달이 보장되지 않으므로 페널티는 관대하게 운영한다. (예: 첫 1~2회는 안내/경고만, 3회부터 실질 페널티.) 값과 강도는 파일럿 피드백으로 조정.

**(d) 최소 세션 필터 [확정]**
- **10분(값 튜닝 가능) 미만 세션은 집계에 반영하지 않는다**(랭킹·통계·캘린더 전부). 입실 직후 바로 퇴실하는 조작 방지.
- **적용 위치 = 집계 단계**: 10분 미만 세션은 `daily_study_records` 재계산 시 아예 제외한다. (초안처럼 랭킹 쿼리 `HAVING SUM >= 10분`으로 걸면 "5분짜리 세션 20번" 같은 케이스를 못 거른다 — 그건 기간 총합 필터라 의미가 다르다.)
- **판정 기준 = 분할 전 세션 전체 길이**(`check_out_at - check_in_at`). 04:00을 걸친 정당한 긴 세션의 경계 세그먼트(짧을 수 있음)를 세그먼트 단위로 버리지 않기 위함.

**(e) 랭킹 시간 캡 — 하루 16h + 주 84h [확정]**
- **대원칙: 기록은 실제 시간 그대로 저장하고, 캡은 "랭킹 계산 시에만" 씌운다.** 따라서 `daily_study_records`·마이페이지·캘린더는 **언제나 실제 공부시간(예: 23h) 그대로** 표시하고, 캡은 랭킹 집계 쿼리에서만 적용한다.
- **하루 16h 캡 (A안 — 초과분만 제외)**: 랭킹 합산 시 각 스터디 날짜의 하루 합을 `LEAST(하루합, 16h)`로 자른 뒤 기간 합산한다. 즉 하루 16h까지는 랭킹에 인정하고 그 이상만 잘린다. (예: 23h 공부 → 마이페이지 23h, 랭킹 반영 16h. 15h59m은 그대로 인정.)
  - "16h 넘으면 그날 통째로 0"인 B안은 **채택하지 않음** — 15h59m은 다 인정되고 16h01m은 0이 되는 절벽(cliff)이 생겨 더 열심히 한 사람을 역전 처벌하기 때문.
  - 목적: 05:00 체크인 후 미체크아웃 같은 **단일 유령 세션(하루 ~23h)의 한 방 뻥튀기**를 16h로 트림.
- **주 84h 캡 (=12h×7, 2차 백스톱)**: 주간 랭킹에서 위 하루-캡 합계가 84h를 넘으면 84h로 한 번 더 캡. 하루 16h씩 매일(주 112h) 같은 비현실적 누적을 차단.
- 역할 분담: **하루 캡 = "한 방 사고" 방어 / 주 캡 = "누적 사고" 방어**(겹치지 않음). 04:00 자동 마감(§3.6a)이 세션을 날짜 안에 묶는 0차 방어.
- 값(하루 16h / 주 84h)은 §9 조정 가능.

**(f) (선택) 세션당 상한**
- 04:00 자동 마감으로 세션은 이미 <24h로 묶이지만, 더 죄고 싶으면 **세션당 상한(예: 18h)**을 추가해 초과 시 `FORCE_CLOSED` 처리할 수 있다. 파일럿 기본은 미적용(§12).

**(g) [미확정] 카페 마감시각 강제 체크아웃**: 대상 카페가 24시간 운영이라 기본적으로 불필요. 향후 영업시간이 고정된 카페로 확장 시에만 고려 (§12).

### 3.7 [확정] 무소속 처리

- 학교는 회원가입 시 설정한다. 성인 이용자 등 학교가 없는 사용자는 **무소속**(`school_id = null`)으로 등록.
- **무소속 사용자는 개인별 랭킹에만 등장**하고, 학교별 랭킹과 "우리 학교 페이지"(§5.3)에는 포함되지 않는다.
- **[설계] 학교는 미리 정의된 목록에서 선택**하게 한다(자유 입력 금지). 자유 입력이면 "서울대"/"서울대학교" 같은 표기 분산으로 집계가 깨진다. → `schools` 테이블 + 회원가입 시 검색/드롭다운 선택.
- **학교 소속 검증은 하지 않는다(파일럿).** 운영자가 주변/자주 오는 학교 + 무소속을 큐레이션해 목록으로 제공하고, 사용자는 본인 학교를 찾거나 없으면 무소속을 선택한다. 소속 위조 가능성은 수용하는 한계로 §8에 명시.

## 4. 데이터 모델

> 컬럼은 핵심만 표기. 타입/제약은 JPA 엔티티로 구체화. 시간은 모두 `timestamptz`(UTC 저장), 애플리케이션에서 KST 기준으로 스터디 날짜 계산.

### `schools`
- `id` (PK)
- `name` (unique)
- `short_name` (마스킹 표기용 축약명, 예: "ㅁㅁ중". null이면 `name` 사용)
- `created_at`

### `users`
- `id` (PK)
- `login_id` 또는 `email` (unique, 로그인 자격 — 둘 중 하나로 통일)
- `password_hash` (BCrypt 등)
- `display_name` (실명 — 랭킹에서 마스킹되어 표시)
- `name_seq` (int, default 1 — 같은 학교 동명이인 접미 숫자. 1은 미표기)
- `school_id` (FK, nullable → 무소속)
- `created_at`

### `cafes` (확장성 위해 분리, 파일럿은 1행)
- `id` (PK)
- `name`
- `qr_token` (카페 QR에 담기는 식별 토큰)
- `open_time` / `close_time` (24h 카페라 파일럿 미사용, 향후 마감 강제 종료용 선택 컬럼)
- `created_at`

### `check_in_sessions` (원본 기록 = 진실의 원천)
- `id` (PK)
- `user_id` (FK)
- `cafe_id` (FK)
- `check_in_at` (timestamptz)
- `check_out_at` (timestamptz, 활성 중엔 null)
- `status` (ENUM: `ACTIVE` / `COMPLETED`(수동 체크아웃) / `AUTO_CLOSED`(04:00 자동 마감) / `FORCE_CLOSED`(선택적 세션당 상한 초과))
- `created_at`
- **제약: 유저당 `ACTIVE` 세션은 최대 1개.** Postgres 부분 유니크 인덱스로 강제: `UNIQUE(user_id) WHERE status='ACTIVE'`. (토글 더블탭 레이스로 세션 2개/이중 체크아웃 생기는 것 방지.)
- 인덱스: `(user_id, status)`(활성 조회), `(check_out_at)`(배치·tie-break).

### `daily_study_records` (집계 — 04:00 분할 결과)
- `id` (PK)
- `user_id` (FK)
- `study_date` (date, 04:00 기준 스터디 날짜)
- `total_seconds` (int, 해당 유저·해당 스터디날짜의 유효 세션 세그먼트 합. §3.6d 최소 세션 필터 통과분만). **실제 시간 그대로 저장 — 하루 16h/주 84h 랭킹 캡(§3.6e)은 저장값이 아니라 랭킹 조회 시점에만 적용한다.**
- **UNIQUE(`user_id`, `study_date`)**
- `updated_at`
- 인덱스: `(study_date)`, `(user_id, study_date)`(UNIQUE로 커버).
- 갱신은 **세션이 닫힐 때 해당 날짜를 원본에서 재계산해 SET**(멱등, §3.1).
- → 캘린더, 주간·월간 합계, 요일별 통계, 개인·학교 랭킹의 기반.

### `user_warnings` (경고 이력 — §3.6c)
- `id` (PK)
- `user_id` (FK)
- `session_id` (FK, 어떤 자동 마감 세션에서 발생했는지)
- `reason` (ENUM/문자열: 예 `AUTO_CLOSED_AT_0400`)
- `created_at`
- → 최근(당월) 경고 수를 세어 페널티 판정. 매월 리셋은 "당월 `created_at` 카운트"로 자연 처리 가능.

### `push_subscriptions` (선택 — Web Push 사용 시)
- `id` (PK)
- `user_id` (FK)
- `endpoint`, `p256dh`, `auth` (Web Push 구독 정보)
- `created_at`
- → 03:30 사전 알림 전송 대상. (Web Push 미도입 시 이 테이블·§6 구독 API 생략.)

### `hourly_study_pattern` (선택 — 24시간 히스토그램, 전체 기간 누적)
- `user_id` (FK)
- `hour_of_day` (0~23, 벽시계 시각 기준, 04:00 무관)
- `total_seconds` (int, 누적)
- **UNIQUE(`user_id`, `hour_of_day`)**
- → 24시간 히스토그램은 이 테이블 유지 또는 원본 세션에서 온디맨드 계산 중 택1. **파일럿 규모(수십 명)에선 온디맨드로 충분.** 유지 시엔 세션 종료에서 시(hour) 경계로 분할해 누적.

## 5. 화면 명세

앱은 크게 **마이 페이지 / 우리 학교 페이지 / 랭킹 페이지**. (+ 회원가입/로그인, 체크인)

### 5.1 마이 페이지 — 본인 공부 기록

- **[확정] 이번 주 총 공부시간**, **이번 달 총 공부시간** (스터디 날짜 기준 §3.2). **실제 공부시간 그대로 표시 — 하루 16h/주 84h 랭킹 캡(§3.6e)은 적용하지 않는다.**
- **[확정] 요일별 공부 패턴** — "그동안 누적된 요일별 평균"(예: "평균적으로 화요일에 가장 많이 공부"). 이번 주 breakdown이 아니라 **누적 평균 패턴**. (재미 요소)
  - **[설계]** 각 요일에 대해, 기록이 있는 해당 요일 스터디날짜들의 하루 공부시간 평균. (단순 해석, 조정 가능)
- **[확정] 24시간 시간대별 공부량** — 몇 시에 많이 공부하는지. **벽시계 시각(0~23시) 그대로 버킷**, 04:00 분할 미적용 (§3.2 미적용 대상).
- **[확정] 월간 캘린더** — 각 날짜 칸에 그날 공부 시간을 `HH시간 MM분` 형식으로 표시.
  - 하루에 여러 번 체크인/체크아웃했으면 **합산** 표시 (§3.1의 스터디 날짜 기준 합계).
- **현재 상태**: 활성 세션이 있으면 "HH:MM부터 공부 중" 표시. (§6 `sessions/current`)

### 5.2 랭킹 페이지

**상단 컨트롤 (2단):**
1. **[확정] 랭킹 종류 탭**: `개인별 랭킹` / `학교별 랭킹` — 터치로 전환, 화면의 랭킹이 즉시 바뀜.
2. **[확정] 기간 필터**: 기본값 **이번주**. 버튼을 누르면 다른 기간 옵션이 펼쳐지고, 선택 시 해당 기간 랭킹으로 갱신.
   - **[확정] 과거 기간 조회 포함.**
   - **[설계] 옵션 세트**: `이번주`(기본) / `지난주` / `이번달` / `지난달` / `올해`. (§12 조정 가능)

**개인별 랭킹:**
- **[확정] 1~3위 단상(포디움)**: 상단에 크게. 표시 형식 `김O현(ㅁㅁ중) HH시간 MM분` (이름 마스킹+학교 §3.3).
- **[확정] 4~10위**: 그 아래 차트/리스트 형태.
- **[확정] 10위 밖 사용자를 위한 "내 순위"**: 리스트 하단에 본인 순위와 시간을 항상 표시 (예: `내 순위: 37등 · 12시간 30분`). Top 10에 들면 리스트 안에서 강조 표시.
  - **[확정] 해당 기간에 기록이 없으면 "순위 없음 / 기록 없음"으로 표시** (0등이 아니라 미랭크).
- 포디움 참여자가 3명 미만이면 있는 만큼만 표시.

**학교별 랭킹:**
- **[확정] 1~3위 단상**: 개인과 동일한 포디움 형태.
- **[확정] 평균 기준 + 인원수 표시** (§3.4): 예 `서울대학교 · 24명 · 평균 18시간 30분`.
- **[확정] 최소 인원(5명) 미달 학교는 노출 제외** (§3.4).
- 동점 처리 §3.5. 무소속은 학교 랭킹에 미등장 (§3.7).

### 5.3 우리 학교 페이지 [확정 · 신설]

- **같은 학교 안에서의 개인 공부시간 랭킹**을 보여준다(마이페이지 옆). 학교 내부 경쟁/소속감 요소.
- 구성은 개인별 랭킹과 동일(포디움 1~3, 리스트 4~10, "내 순위" 행, 기간 필터)하되 **대상 = 내 `school_id` 소속 학생으로 한정**. 이름은 동일하게 마스킹.
- 기존 개인 랭킹 로직에 `WHERE school_id = :myschool` 필터만 추가하면 되므로 구현 부담이 작다.
- **무소속 사용자**는 이 페이지 대신 안내 문구("학교를 설정하면 우리 학교 랭킹을 볼 수 있어요") 표시.

## 6. API 명세 (개요)

> 경로/응답 스키마는 예시. 구체 필드는 구현 시 확정. 모든 통계·랭킹·체크인 엔드포인트는 JWT 필요.

### 인증
- `POST /api/auth/signup` — `{ loginId, password, displayName, schoolId? }`. 가입 시 같은 학교 동명이인이면 `name_seq` 부여(§3.3).
- `POST /api/auth/login` — `{ loginId, password }` → JWT
- `GET  /api/schools` — 회원가입용 학교 목록(검색/드롭다운)
- `GET  /api/users/me` — 현재 사용자 정보(경고 수/페널티 상태 포함 가능)

### 체크인 / 체크아웃
- `POST /api/sessions/toggle` — `{ cafeToken }` (QR에서 추출).
  - **[설계] 토글 방식**: 활성 세션이 있으면 체크아웃(`COMPLETED`), 없으면 체크인. 카페는 **QR 한 장**만 부착. 체크아웃 시 §3.1 분할 → `daily_study_records` 재계산 upsert.
  - **동시성**: 트랜잭션 + `ACTIVE` 부분 유니크 인덱스(§4)로 더블탭 레이스 방지.
- `GET  /api/sessions/current` — 현재 활성 세션(있으면 "HH:MM부터 공부 중" 표시용)

### 마이 페이지 통계
- `GET /api/me/stats/overview` — 이번 주 / 이번 달 총합 (**실제 시간 그대로, 랭킹 캡 §3.6e 미적용** — 마이페이지이므로)
- `GET /api/me/stats/calendar?year=&month=` — 해당 월 날짜별 합계(초/분)
- `GET /api/me/stats/weekday-pattern` — 요일별(7개) 누적 평균
- `GET /api/me/stats/hourly-pattern` — 시간대별(24개) 누적

### 랭킹
- `GET /api/rankings/individual?period=this_week|last_week|this_month|last_month|this_year` — 순위 리스트 + 내 순위
- `GET /api/rankings/school?period=...` — 학교 순위(평균, 활동 인원수), 최소 인원 필터 적용
- `GET /api/rankings/school/mine?period=...` — **우리 학교 내부 개인 랭킹**(§5.3). 내 `school_id` 한정. 무소속이면 빈 결과/안내.

### 알림 (Web Push 도입 시)
- `POST /api/push/subscribe` — `{ endpoint, keys:{p256dh, auth} }` 저장
- (전송은 서버 배치가 담당, §스케줄러)

### 개인 랭킹 쿼리 개념 (참고 — 최소 세션 필터는 이미 집계에서 처리됨)
```sql
-- 하루 16h 캡을 먼저 씌운 뒤 기간 합산 (A안, §3.6e)
SELECT u.id, u.display_name, u.school_id,
       SUM(LEAST(d.total_seconds, 16*3600)) AS total   -- 하루 16h 캡(초과분만 제외)
FROM daily_study_records d
JOIN users u ON u.id = d.user_id
WHERE d.study_date BETWEEN :start AND :end
GROUP BY u.id, u.display_name, u.school_id
ORDER BY total DESC,
         /* 동점: 기간 내 마지막 체크아웃 이른 순 */ last_checkout_in_period ASC;
-- 최소 세션(10분) 필터는 daily_study_records 생성 시 이미 반영됨 → HAVING 불필요.
-- 주간 랭킹은 위 total을 LEAST(total, 84*3600)로 한 번 더 캡(주 84h 백스톱).
-- 페널티 대상자 제외는 애플리케이션/조건에서 처리(§3.6c).
-- ⚠️ 마이페이지 총합/캘린더는 캡 없이 SUM(d.total_seconds) = 실제 시간 그대로.
```
- 동점 tie-break용 `last_checkout_in_period`는 `check_in_sessions`에서 기간 내 유저별 `MAX(check_out_at)`로 구한다. (세션이 두 스터디날짜에 걸치는 경계 케이스의 정밀도는 사소하므로 합리적 구현이면 됨.)
- 학교 랭킹: 위 개인 합계를 `school_id`로 묶어 **평균 = SUM/활동인원**, `HAVING 활동인원 >= 5`, 동점은 활동 인원 desc → 학교명 asc (§3.5).

### 스케줄러(배치)
- **04:00 일일 자동 마감**(§3.6a): `ACTIVE` 세션 `AUTO_CLOSED`(check_out=04:00) → 분할·집계 → 해당 유저 경고 1 적립.
- **03:30 사전 알림**(§3.6b): `ACTIVE` 유저에게 Web Push.
- (선택) 세션당 상한 초과 `FORCE_CLOSED`(§3.6f).

## 7. 단계별 구현 로드맵

> 원칙: **각 Phase를 구현·테스트한 뒤 다음으로.** Phase마다 명확한 메시지로 커밋. 백엔드가 사용자의 강점이므로 백엔드 우선, 정확성이 중요한 집계 로직을 앞쪽에 배치.

- **Phase 0 — 프로젝트 셋업**
  - **모노레포**(`backend/`, `frontend/`) 단일 git 레포, GitHub **public** 생성, `.gitignore`로 시크릿(+VAPID) 제외
  - 로컬 Postgres(또는 Docker) 준비
  - Spring Boot 초기화(Web, JPA, Security, Validation, PostgreSQL)
  - Vite + React + `vite-plugin-pwa` 초기화

- **Phase 1 — 인증(백엔드)**
  - `users`, `schools` 엔티티 / 회원가입·로그인 / JWT 발급·검증 / 비밀번호 해싱 / `GET /api/schools`
  - 동명이인 `name_seq` 부여 로직
  - HTTP 클라이언트(Postman 등)로 검증

- **Phase 2 — 체크인/체크아웃 원본 기록(백엔드)**
  - `cafes`, `check_in_sessions` / `toggle`·`current` 엔드포인트
  - **`ACTIVE` 부분 유니크 인덱스 + 토글 트랜잭션**(동시성)
  - 이 단계에선 집계 없이 **원본 세션만** 정확히 기록

- **Phase 3 — 집계 로직(백엔드, ⚠️ 최우선 정확성)**
  - 세션이 닫히는 **모든 경우**(수동/자동/강제)에 §3.1 분할 → `daily_study_records` **재계산 SET(멱등)**
  - 최소 세션(10분) 필터를 집계 단계에서 적용
  - (선택) `hourly_study_pattern` 누적
  - **§3.1 검증 예시(8일 18:00~9일 06:00 → 10h/2h)를 반드시 단위 테스트로 통과.** 정각 경계, 자정 무포함/포함, 다중 세션 합산, 재처리 멱등성(같은 세션 두 번 처리해도 총합 불변), 10분 필터 케이스도 테스트.

- **Phase 4 — 마이페이지 통계 API(백엔드)**
  - overview(84h 상한 포함) / calendar / weekday-pattern / hourly-pattern

- **Phase 5 — 랭킹 API(백엔드)**
  - 개인 랭킹(내 순위 포함, 동점 처리, 미랭크 처리) / 학교 랭킹(평균 + 최소인원 5 + 활동인원수 + 동점) / 우리 학교 내부 랭킹

- **Phase 6 — 프론트 기초(PWA)**
  - 라우팅(마이/우리학교/랭킹), 회원가입(학교 선택)·로그인 화면, JWT 저장 및 요청 첨부, manifest·설치

- **Phase 7 — 체크인 화면(프론트)**
  - `html5-qrcode`로 QR 스캔 → `toggle` 호출 → 현재 공부 상태 표시. "04:00 이후 재체크인" 안내 상시 노출.

- **Phase 8 — 마이페이지(프론트)**
  - 요약 카드, 캘린더(칸별 `HH시간 MM분`), 요일 차트, 시간대 차트

- **Phase 9 — 랭킹/우리학교 페이지(프론트)**
  - 종류 탭(개인/학교), 기간 필터(기본 이번주 + 펼침), 포디움(1~3), 리스트(4~10), 내 순위 행(미랭크 처리), 이름 표기 `김O현(ㅁㅁ중)`, 우리 학교 페이지

- **Phase 10 — 마감/어뷰징 방지/스케줄러**
  - **04:00 일일 자동 마감 배치**(분할·집계·경고 적립), **03:30 사전 알림 배치**, 경고 누적→페널티, 주간 84h 상한, (선택) 세션당 상한
  - Web Push 구독(`push/subscribe`) + service worker `push` 처리. **도달 한계(§8) 감안해 페널티는 관대하게.**

- **Phase 11 — 실기기 테스트**
  - iPhone Safari + Android Chrome에서 카메라/PWA 설치/**Web Push(설치 PWA·iOS16.4+)** 동작 확인, 04:00/03:30 배치가 배포 환경에서 정확히 도는지 확인, 플랫폼별 차이 보정

- **Phase 12 — 배포(추후 확정)**
  - 호스팅 선정 → 백엔드 + Postgres + 프론트 정적 배포 → HTTPS 도메인 연결 → 스케줄 배치/슬립 이슈 점검 → 실기기 재검증

## 8. 정직하게 알아둘 한계 (설계에 반영됨)

1. **물리적 착석 검증 불가**: 정적 QR 스티커는 사진을 찍어 원격 체크인이 가능하다. 파일럿은 이를 수용한다. (향후 강화: 태블릿에 회전 QR 표시 방식 — 파일럿 미구현, §12)
2. **미체크아웃 시간 인플레이션은 04:00까지 발생 가능**: 04:00 자동 마감이 상한을 04:00으로 묶지만, 예컨대 23:00에 나가고 체크아웃을 잊으면 다음 04:00까지의 시간이 잘못 인정될 수 있다(최대 몇 시간). 완전 차단은 불가하며, 경고/페널티(§3.6c)로 반복 미체크아웃을 억제한다.
3. **밤샘 이용자 vs 미체크아웃 구분 불가**: 04:00에 켜져 있는 세션이 "진짜 밤샘"인지 "깜빡"인지 시스템은 구분할 수 없다. 그래서 04:00 자동 마감은 **모두에게 동일 적용**하고, 계속할 사람은 04:00 이후 재체크인하는 것으로 정리한다. 03:30 사전 알림이 이 안내를 돕는다.
4. **Web Push 도달 한계**: iOS는 **홈 화면에 설치한 PWA + iOS 16.4+**에서만 푸시가 온다. 미설치/구형/권한 미허용 사용자는 03:30 알림을 못 받는다. 따라서 알림은 베스트 에포트이고, 재체크인 안내는 앱 내에도 상시 노출하며, 페널티는 관대하게 운영한다.
5. **학교 소속 위조 가능**: 가입 시 학교를 검증 없이 선택하므로, 라이벌 학교를 골라 평균을 흔들거나 소속을 사칭할 수 있다. 파일럿은 이를 수용하며(운영자 큐레이션 목록 + 최소 인원 필터로 완화), 필요 시 향후 학교 이메일/코드 검증을 검토한다.
6. **자동 체크아웃 없음**: 웹 표준 한계로 백그라운드 자동 체크아웃 불가 — 04:00 자동 마감이 유일한 안전망이다.
7. **무료 호스팅 슬립/콜드스타트**: 유휴 후 첫 요청 지연 및 **스케줄 배치 실행 신뢰성** 문제가 생길 수 있다. 배포 단계에서 트레이드오프/대안(외부 크론 트리거 등) 판단.

## 9. 미확정 결정 사항 (사용자 확인 필요 — 기본값으로 진행 가능)

각 항목은 위 명세에 기본값이 반영되어 있으므로 구현은 진행할 수 있고, 이 부분만 쉽게 교체 가능하게 만든다.

1. **학교 최소 인원 기준값**: 기본 **5명**.
2. **최소 세션 필터 값**: 기본 **10분**.
3. **랭킹 시간 캡**: 기본 **하루 16h + 주 84h**(랭킹에만 적용, 마이페이지는 실제 시간 / 초과분만 제외 = A안, §3.6e). (선택) **세션당 상한**: 기본 미적용(적용 시 18h 제안).
4. **03:30 사전 알림 시각**: 기본 **03:30**(03:00~03:30 조정 가능).
5. **경고→페널티 임계치**: 기본 **3회**. **페널티 강도**: 기본 "해당 주/쿨다운 랭킹 제외 + 프로필 표시"(더 약하게/강하게 조정 가능). **경고 리셋 주기**: 기본 **월간**.
6. **Web Push 도입 여부**: 도입 시 `push_subscriptions` + VAPID 필요. 미도입 시 03:30 알림은 앱 오픈 상태 인앱 알림으로 축소.
7. **기간 옵션 세트**: 이번주/지난주/이번달/지난달/올해.
8. **학교 랭킹 기간 정책**: 기본은 개인과 동일한 기간 세트 + "선택 기간의 활동 인원" 분모. (월간만 노출하고 싶으면 조정.)
9. **`display_name` = 실명** 전제(마스킹용). 닉네임 미사용, 표기 `김O현(ㅁㅁ중)`.
10. **로그인 자격**: `login_id` vs `email` 중 택1.
11. **원격 체크인 방지(회전 QR)**: 파일럿 미구현, 향후 과제.
12. **카페 마감시각 강제 체크아웃**: 24h 카페라 미적용. 향후 확장 시 고려.
